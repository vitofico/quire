# Quire AI: book insights and library intelligence

Date: 2026-05-09
Status: Draft (design)

## Problem

Quire today is a privacy-first reader. Calibre-web is the catalog, opds-sync
is the reading-state server, and the README states plainly: *"Network calls
go to exactly two places: your calibre-web instance and your opds-sync server.
No analytics, no crash reporting, no third-party SDKs."*

Users have asked for AI-assisted features on top of the existing stack:

- **Book insights** вЂ” meaningful, grounded information about a book the user is
  about to read: who the author is, what the book is, what series it's part
  of, themes, tone, content advisories. Static (generated once, cached
  forever) for v1; paragraph-level discussion and AI-assisted note-taking
  later.
- **Library intelligence** вЂ” analysis of the user's library: a reader profile
  derived from reading history, suggestions for what to read next from the
  books they already own, and suggestions for what to acquire next.

The work needs to add an AI substrate to the project without sacrificing the
project's stance: optional, opt-in, configurable to point at a self-hosted
endpoint (Ollama, llama.cpp, vLLM), and honest about exactly what data leaves
the homelab.

## Goals

- An AI substrate in opds-sync that is OpenAI-compatible (one client, two
  audiences: cloud BYOK and self-hosted).
- A `book_insights` cache, keyed by document, shared across all of a user's
  devices and across users on the same instance.
- Lazy generation on book-detail open, with server-side coalescing so two
  devices opening the same book at once trigger one model call.
- Deterministic external grounding from Wikipedia and OpenLibrary before the
  model is called, so insights are citable and small local models stay useful.
- Library-wide analysis: reader profile + recommendations from owned books +
  suggestions for new acquisitions.
- Per-user opt-in for AI features. The admin enables AI server-wide; each
  user explicitly opts in from Quire settings.
- F-Droid-clean Android client. No closed-source SDKs, no new permission, no
  bundled AI library вЂ” only HTTP against the existing opds-sync.

## Non-goals (v1)

- Paragraph-level Q&A and AI-assisted note-taking. The substrate must not
  preclude them; the features are out of scope here.
- Open-ended web search via tool-calling. Wikipedia + OpenLibrary cover the
  v1 grounding need; general web search via Brave/SearXNG/etc. is a later
  opt-in.
- Per-user AI provider keys. Server-wide config only.
- Background pre-generation of insights at download time, or nightly profile
  refresh. v1 generates on demand.
- Localization of insight text. v1 is English only; the schema leaves room
  to grow.
- Quotas, billing, per-user rate limits. Server-wide concurrency cap is
  enough.
- ProGuard/R8 changes specifically motivated by AI; tracked separately.

## Decisions

| # | Decision | Rationale |
|---|---|---|
| 1 | OpenAI-compatible chat-completions client, configured by `base_url` + `api_key` + `model`. | One client serves both Ollama (homelab default) and any cloud BYOK provider; users pick where their data goes. |
| 2 | All AI calls originate from opds-sync, not from the Android app. | Library data is server-side; cache is shared across devices and users; one config; coalescing; long-running jobs survive app death. |
| 3 | Server-wide AI config in opds-sync env. No per-user keys. | Matches the homelab audience; avoids encrypting per-user secrets in Postgres; per-user keys can be added later without breaking shape. |
| 4 | Insight payload is structured JSON, not free-form Markdown. | Forces the model to admit ignorance per-field instead of confabulating; lets the UI render per-section cards; cleaner cache and migration story. |
| 5 | Lazy generation on book-detail open. | Cache means generation is paid once per book ever; the canonical Ollama deploy pays effectively zero per token. |
| 6 | Prompt is fueled by title + author + EPUB OPF metadata bundle. | The publisher blurb in `dc:description` and Calibre's `belongs-to-collection` series field are strong, deterministic anchors. Series info bypasses the model entirely when the OPF has it. |
| 7 | Deterministic retrieval from Wikipedia + OpenLibrary before the model call. | Grounds the model in citable sources; makes small local models as good as cloud for the common case; per-source toggle in admin config; results cached server-side and reusable across books by the same author. |
| 8 | Per-user opt-in toggle, off by default. | Privacy-positioned project; the toggle is also the place to disclose what gets sent and where. Cached insights generated by an opted-in user are still readable by opted-out users (cache is content-addressed). |
| 9 | Generation is coalesced server-side via in-process per-document async locks. | Single-replica deployment is the canonical case; in-process locks are sufficient. Multi-replica scaling is YAGNI. |
| 10 | Cache key includes `model_id` and `prompt_version`. Bumping either implicitly invalidates without deleting old rows. | Lets us iterate prompts without a destructive migration; old generations stay queryable for diagnostic purposes. |
| 11 | Reader profile and recommendations are persisted per-user; refreshed only on explicit user request. | Reading history changes slowly; recompute-on-open is wasteful; explicit refresh keeps the user in control of when reading history is shipped to the model. |

## Server design

### Config

New env vars on opds-sync (additions to `opds_sync/config.py`):

| Var | Default | Notes |
|---|---|---|
| `OPDS_SYNC_AI_ENABLED` | `false` | Master switch. When false, all `/ai/v1/*` endpoints return 503 except `GET /ai/v1/config`, which reports `configured: false`. |
| `OPDS_SYNC_AI_BASE_URL` | _(none)_ | OpenAI-compatible base URL. Required if `AI_ENABLED`. Example: `http://ollama.lan:11434/v1`. |
| `OPDS_SYNC_AI_API_KEY` | _(none)_ | Bearer token. Optional for Ollama. |
| `OPDS_SYNC_AI_MODEL` | _(none)_ | Model identifier passed verbatim. Required if `AI_ENABLED`. |
| `OPDS_SYNC_AI_TIMEOUT_S` | `120` | Single-request timeout for the AI call. |
| `OPDS_SYNC_AI_MAX_CONCURRENCY` | `4` | Server-wide cap on concurrent AI calls. Implemented as `asyncio.Semaphore`. |
| `OPDS_SYNC_AI_SOURCES` | `wikipedia,openlibrary` | CSV of enabled retrieval sources. Empty string disables all retrieval. |
| `OPDS_SYNC_AI_RETRIEVAL_TIMEOUT_S` | `8` | Timeout per external lookup. Lookup failures are non-fatal; the model is called with the retrieval results that did succeed. |
| `OPDS_SYNC_AI_PROMPT_VERSION` | `"1"` | Bumped in code when prompts change; included in cache key. Exposed via env so deployers can pin a specific version. |

`OPDS_SYNC_AI_API_KEY` is never logged and is never returned by any endpoint.

### Schema

Four new tables. Migration in `server/migrations/`.

`book_insights` вЂ” content-addressed cache, *not* keyed on the per-user
`documents` table. The existing `Document` rows are user-scoped (a row per
user per book); insights are a global resource keyed on document identity:

```
id              bigserial    PK
metadata_id     text         NULL          -- normalized; same rules as core/identity
content_hash    text         NOT NULL      -- always present
model_id        text         NOT NULL
prompt_version  text         NOT NULL
sources_used    text[]       NOT NULL      -- e.g. {wikipedia,openlibrary}
payload         jsonb        NOT NULL      -- structured insight schema (below)
sources         jsonb        NOT NULL      -- citation list: [{title,url,snippet,kind}]
generated_at    timestamptz  NOT NULL DEFAULT now()
generated_by    text         NOT NULL      -- user_id of the user who triggered

UNIQUE (metadata_id, model_id, prompt_version)        -- partial: WHERE metadata_id IS NOT NULL
UNIQUE (content_hash, model_id, prompt_version)       -- always
INDEX  (content_hash)
INDEX  (metadata_id) WHERE metadata_id IS NOT NULL
```

Lookup precedence mirrors `core/identity` В§"Lookup precedence": prefer
`metadata_id` when the request has one; fall back to `content_hash`.

Alias reconciliation (a row written under `content_hash` only, then later
matched by a request that also carries `metadata_id`) backfills the row's
`metadata_id` column in one transaction. If two pre-existing rows would
collide on the new `(metadata_id, model_id, prompt_version)`, the older
generation is deleted in favor of the newer.

`reader_profiles` вЂ” per-user, single current profile:

```
user_id         text         PK
payload         jsonb        NOT NULL  -- profile + recommendations schema
model_id        text         NOT NULL
prompt_version  text         NOT NULL
generated_at    timestamptz  NOT NULL DEFAULT now()
```

`user_ai_preferences`:

```
user_id     text         PK
ai_enabled  boolean      NOT NULL DEFAULT false
updated_at  timestamptz  NOT NULL DEFAULT now()
```

`external_source_cache` вЂ” deterministic retrieval cache:

```
source     text         NOT NULL  -- "wikipedia" | "openlibrary"
key        text         NOT NULL  -- normalized lookup key (see retrieval)
payload    jsonb        NOT NULL
fetched_at timestamptz  NOT NULL DEFAULT now()

PRIMARY KEY (source, key)
```

Retrieved entries older than 30 days are refetched on next use (lazy
expiration; no nightly GC needed).

### Insight payload schema

Stored as JSON in `book_insights.payload`. Pydantic model in
`opds_sync/api/ai_schemas.py`. Every field can be `null`; the UI renders
only the present ones.

```python
class BookInsight(BaseModel):
    schema_version: int = 1
    summary: str | None              # 2вЂ“4 sentence description of the book
    author: AuthorInsight | None     # bio + notable_works
    series: SeriesInsight | None     # name + position + total_known
    themes: list[str] | None         # short tags, e.g. ["dystopia", "first contact"]
    tone: str | None                 # one short phrase
    content_advisory: list[str] | None  # e.g. ["graphic violence"]
    suggested_for: str | None        # one short phrase: "readers who liked X"
    confidence: Literal["high","medium","low"]  # model's self-assessment
    notes: str | None                # anything noteworthy not captured above

class AuthorInsight(BaseModel):
    bio: str | None
    notable_works: list[str] | None
    nationality: str | None
    active_years: str | None         # "1949вЂ“1992"

class SeriesInsight(BaseModel):
    name: str
    position: int | None
    total_known: int | None
```

Series is populated deterministically from the OPF `belongs-to-collection`
when present; otherwise the model fills it. Series sourced from the OPF is
marked in `sources` with `kind: "opf"`.

### Reader profile + recommendations payload

```python
class ReaderProfile(BaseModel):
    schema_version: int = 1
    stats: ReaderStats               # deterministic, computed from Postgres
    narrative: str                   # 3вЂ“6 sentence model-written summary
    in_library_recommendations: list[BookRec]   # books user already owns
    discovery_recommendations: list[BookRec]    # books user doesn't own
    confidence: Literal["high","medium","low"]

class ReaderStats(BaseModel):
    total_books: int
    finished_count: int
    abandoned_count: int       # opened but not progressed for >90 days
    in_progress_count: int
    avg_session_minutes: float | None
    finish_rate_by_theme: dict[str, float]   # derived from cached insights
    most_read_authors: list[str]

class BookRec(BaseModel):
    title: str
    author: str
    identity: DocumentIdentity | None  # set for in-library recs (so Quire can link to the existing book screen); null for discovery
    rationale: str             # 1вЂ“2 sentences, model-written
    sources: list[Citation] | None
```

Discovery recommendations cite OpenLibrary author bibliographies where
possible (so the user can verify the suggested book exists). Quire shows
"AI-suggested" in the UI for any rec without authoritative citations.

### Endpoints

All endpoints under `/ai/v1/`, behind the existing calibre-web Basic auth
proxy. Opt-in checked per request unless noted.

| Method | Path | Notes |
|---|---|---|
| `GET` | `/ai/v1/config` | Public to authed users. Returns `{configured, base_url_host, model_id, sources_enabled}`. No auth-state enforcement on opt-in (the app needs this to render the toggle). |
| `GET` | `/ai/v1/preferences` | Returns `{ai_enabled}` for the calling user. |
| `PUT` | `/ai/v1/preferences` | Body `{ai_enabled}`. Sets and returns. |
| `POST` | `/ai/v1/insights/lookup` | Body: `{identity: DocumentIdentity, bundle: MetadataBundle}`. On cache hit returns immediately. On miss generates synchronously, persists, returns. Requires opt-in. |
| `POST` | `/ai/v1/insights/get` | Body: `{identity: DocumentIdentity}`. Cache-only read. 404 if no entry for the current `(model_id, prompt_version)` under either identity key. Does NOT require opt-in (cached results are shareable). |
| `POST` | `/ai/v1/insights/invalidate` | Body: `{identity: DocumentIdentity}`. Deletes the cache row for the current `(model_id, prompt_version)`. Triggers the next `lookup` to regenerate. Requires opt-in. |
| `GET` | `/ai/v1/profile` | Returns the calling user's profile or 404. Does NOT require opt-in (so the user can read their last profile after opting back out, but no new generation runs). |
| `POST` | `/ai/v1/profile/refresh` | Synchronously regenerates the calling user's profile. Requires opt-in. |

Status codes:

- `200` cached or fresh result.
- `404` no cached entry (GET only).
- `409` user not opted in (POST/DELETE/refresh).
- `502` AI provider failed; the body carries a short error code (`provider_unreachable`, `provider_timeout`, `parse_error`). Quire shows a non-blocking error card.
- `503` AI not configured server-side, or master switch off.

### Metadata bundle (request body)

```json
{
  "title": "Foundation",
  "author": "Isaac Asimov",
  "language": "en",
  "isbn": "9780553293357",
  "publisher": "Bantam Spectra",
  "publish_date": "1991-10-01",
  "subjects": ["Science Fiction"],
  "description": "Publisher blurb from dc:description, if any.",
  "series": { "name": "Foundation", "position": 1 }
}
```

Every field except `title` is optional. The Android side derives this from
the OPF on first download (extending `:core:identity` or adding a thin
`:core:metadata` module that depends on the same OPF parser). The bundle is
*not* persisted on the server вЂ” it travels into the prompt and is forgotten.
If the same document arrives later with a richer bundle, the cached insight
is unaffected; regeneration is explicit (`DELETE` then `POST`).

### Retrieval

`opds_sync/core/ai/retrieval.py` exposes one async function per source:

```python
async def lookup_wikipedia(author: str, title: str) -> list[Citation]: ...
async def lookup_openlibrary(author: str, title: str, isbn: str | None) -> list[Citation]: ...
```

Each:

1. Computes a normalized cache key (lowercase, whitespace-collapsed, ASCII-folded).
2. Reads `external_source_cache`. If a row exists and `fetched_at > now() - 30d`, returns it.
3. Otherwise calls the public API with `OPDS_SYNC_AI_RETRIEVAL_TIMEOUT_S`.
4. Persists and returns. Network failures return `[]` and are logged at `info`; they do not raise.

For Wikipedia: `GET /api/rest_v1/page/summary/{title}` for the book title,
falling back to the author name on miss. Extract the first paragraph as
`snippet`, the canonical URL as `url`.

For OpenLibrary: `GET /search.json?title=...&author=...&limit=3`. For each
hit, fetch the work's `description` and the author's `bio`. Use ISBN when
present to disambiguate.

Both return `Citation` objects:

```python
class Citation(BaseModel):
    kind: Literal["wikipedia","openlibrary","opf","model"]
    title: str
    url: str | None
    snippet: str
```

### AI client

`opds_sync/core/ai/client.py`:

```python
class AIClient:
    async def chat_structured(
        self,
        system: str,
        user: str,
        schema: type[BaseModel],
        *,
        timeout_s: float,
    ) -> BaseModel: ...
```

Talks `POST {base_url}/chat/completions` with `response_format =
{"type": "json_schema", "json_schema": {...}}` when the configured provider
supports it (most do). Falls back to `{"type":"json_object"}` plus a JSON
schema in the system prompt for older endpoints. Parses the result through
the Pydantic model; on `ValidationError` retries once with the validation
error appended to the user message; second failure raises
`ProviderParseError`.

Implementation uses `httpx.AsyncClient` (already a dep). Bearer header from
`OPDS_SYNC_AI_API_KEY`. `User-Agent: opds-sync/<version>`.

### Insight orchestrator

`opds_sync/core/ai/service.py::generate_insight`:

```
1. Fast path: lookup by identity precedence.
     - if `metadata_id` is set: SELECT FROM book_insights
       WHERE metadata_id=? AND model_id=? AND prompt_version=?.
       On hit return.
     - SELECT WHERE content_hash=? AND model_id=? AND prompt_version=?.
       On hit: if `metadata_id` was supplied and the row's `metadata_id`
       is NULL, backfill it (alias reconciliation) and return.

2. Acquire per-identity async lock (in-process dict, key =
   `metadata_id || content_hash`). Re-check the cache (another concurrent
   caller may have just filled it).

3. Acquire global semaphore (OPDS_SYNC_AI_MAX_CONCURRENCY).

4. Run retrieval.lookup_* in parallel for each enabled source.

5. Compose prompt from (metadata bundle, retrieval results, schema description).

6. AIClient.chat_structured(...) with the BookInsight schema.

7. Persist to book_insights with sources from retrieval + the model citation.

8. Release lock + semaphore. Return.
```

Series is populated from the OPF metadata bundle before step 5 if
`belongs-to-collection` was present; the prompt then asks the model not to
override series fields. Author bio is requested but flagged as low-confidence
when no Wikipedia/OpenLibrary author result was found.

### Reader-profile orchestrator

`opds_sync/core/ai/service.py::refresh_profile(user_id)`:

```
1. Compute ReaderStats deterministically from progress + book_insights
   joined on document identity (metadata_id when present, content_hash
   otherwise). Joins to themes/subjects come from cached insights only вЂ”
   profile generation does not trigger new insight generation; books without
   cached insights contribute only their title/author.

2. Build a "reading history digest" for the prompt:
     - last 50 finished books (title, author, finished_at, themes)
     - last 20 abandoned books with abandonment context (% read)
     - 10 most-progressed in-progress books
   No bookmarks, no per-page progress, no snippets.

3. Pull "in-library candidates" вЂ” books the user owns but has not opened вЂ”
   with cached insights joined.

4. Run discovery retrieval: OpenLibrary author bibliographies for the user's
   most-read authors; filter out books they own.

5. Single AIClient.chat_structured call with the ReaderProfile schema.

6. Upsert into reader_profiles. Return.
```

Profile generation is gated by the global semaphore. It's typically one
larger model call rather than a long chain.

## Android design

### Modules

Two changes to the module graph:

- **`:core:metadata`** (new) вЂ” extracts the metadata bundle from an EPUB OPF.
  Depends on the same XML parser already used by `:core:identity`. Pure
  Kotlin/JVM, unit-tested against the same fixture set.
- **`:data:ai`** (new) вЂ” Retrofit/OkHttp client for `/ai/v1/*`. Reuses the
  existing sync `OkHttpClient`+`BasicAuthInterceptor`. JSON serialization
  reuses the same kotlinx-serialization setup as `:data:sync`. DTOs mirror
  the Pydantic schemas above.

`:reader` is unchanged. `:auth` is unchanged. UI surfaces live in `:app`.

### UI surfaces

**Book detail screen** вЂ” three new cards below existing metadata:

- *Summary card.* Renders `payload.summary` and `payload.suggested_for`.
- *Author card.* Renders `AuthorInsight`. Hides empty fields.
- *Series card.* Renders `SeriesInsight` (n/m position).
- *Themes & advisory card.* Tag chips for `themes`; a small advisory
  section when `content_advisory` is present.
- *Sources footer.* "Based on: Wikipedia, OpenLibrary, model `<model_id>`"
  with tappable links. Always shown when the insight has any sources.
- *Overflow menu.* "Regenerate insights" (issues `DELETE` then `POST`).

States:

- *Hidden* if `GET /ai/v1/config` reports `configured: false`, OR the user
  is not opted in.
- *Loading* on first POST (skeleton placeholder for ~5вЂ“60s of generation).
- *Error card* on 502 with a "Retry" button.

**Library screen** вЂ” new "Insights" entry in the library menu:

- "Generate library insights" if no profile exists.
- Reader profile + stats + recommendations otherwise.
- "Refresh" button issues `POST /ai/v1/profile/refresh`. Shows the model id
  and `generated_at` of the current profile.
- "Recommended from your library" cards link to existing book-detail screens.
- "You might want to read next" cards show title, author, rationale, and any
  citations. Explicitly labeled "AI suggestion" with a help affordance.

**Settings screen** вЂ” new "AI features" section:

- Status row: `configured: yes/no`, model id, base-URL host (no key).
- Toggle: "Enable AI features for this account."
- Disclosure copy beneath the toggle:
  > When enabled, Quire sends the title, author, and other EPUB metadata of
  > books you open to **<base_url_host>** (model **<model_id>**) to generate
  > insights. Library analysis additionally sends a digest of your reading
  > history (titles, finished/abandoned status, last-read times). Sources
  > used: **<sources_enabled>**. None of this data is sent until you opt in.

The toggle calls `PUT /ai/v1/preferences`. Off-state is the default for new
users, including users created after the admin enabled AI server-wide.

## Phasing

The work fits one branch but lands in three reviewable chunks.

**Phase 1 вЂ” substrate + book insights v1**
Server: config, schema migration, AIClient, retrieval, insight
orchestrator, `/ai/v1/{config,preferences,insights}` endpoints.
Android: `:core:metadata`, `:data:ai` (insights + preferences), book-detail
cards, settings toggle, disclosure copy.
Ship gate: opted-in user opens an unknown book, sees a generated insight
with at least one citation when `wikipedia` or `openlibrary` is enabled;
opted-out user sees nothing AI-related.

**Phase 2 вЂ” library intelligence**
Server: reader-profile orchestrator, `/ai/v1/profile{,/refresh}` endpoints,
deterministic stats SQL.
Android: library-insights screen, recommendations UI, profile-refresh flow.
Ship gate: a user with в‰Ґ10 finished books gets a coherent profile and
non-empty recommendations from both their owned library and discovery.

**Phase 3 (deferred, not in this design)** вЂ” paragraph Q&A, AI notes,
optional general-web-search tool, per-user provider keys.

## Privacy and disclosure

The README destinations sentence is updated to:

> Network calls go to your calibre-web instance, your opds-sync server, and
> вЂ” if your administrator has enabled AI features and you have opted in вЂ”
> the AI endpoint, Wikipedia, and OpenLibrary.

The Settings disclosure (above) is the single place a user can read exactly
what gets sent. The toggle and disclosure are co-located so consent is
informed at the moment it is given.

The fastlane store description is updated for F-Droid: "AI features
(optional, off by default; require a self-hosted or third-party endpoint
configured by your administrator)."

The opds-sync server logs the AI call as `event=ai_generate
content_hash=<hex> model=<id> latency_ms=<n> sources=<csv>`. It does not log
prompt or completion text.

`/ai/v1/config` exposes only `base_url_host` (the host portion, not the
path), `model_id`, and `sources_enabled`. The API key is never read out.

## Testing

Server (pytest):

- Identity-style fixture set: a handful of EPUB OPF samples в†’ expected
  metadata bundles.
- AIClient against a local fake OpenAI-compatible HTTP server (httpx
  `MockTransport`) covering: structured-output success, validation retry,
  validation final failure в†’ `ProviderParseError`, timeout в†’ `ProviderTimeout`.
- Retrieval against recorded Wikipedia and OpenLibrary fixtures
  (`pytest-recording` or hand-written stubs). Cache hit/miss/expiry paths.
- Orchestrator: cache hit fast path; cache miss happy path; concurrent
  identical requests collapse to one model call (per-doc lock); semaphore
  caps concurrency.
- Endpoints: opt-in gating; 503 when AI disabled; 502 propagation; opt-out
  user can still GET cached insights but cannot POST.
- Profile orchestrator: deterministic stats over a fixture DB; profile
  payload validates against the schema.

Android (JUnit/Robolectric):

- `:core:metadata` extracts the bundle from the same OPF fixtures used on
  the server side, asserting parity (the bundle is the contract).
- `:data:ai` DTOs round-trip JSON for every endpoint.
- Compose UI tests for the book-detail insight cards in each state
  (hidden, loading, loaded with full payload, loaded with minimal payload,
  error, opted-out).
- Settings-toggle test: turning the toggle off triggers `PUT
  /ai/v1/preferences {ai_enabled:false}` and the book-detail cards
  disappear.

End-to-end smoke (dev script, not CI):

- Spin up `docker compose` opds-sync pointed at a local Ollama running
  `llama3.1:8b`. Open three known books via the API; assert all three
  produce a populated insight, with at least one Wikipedia or OpenLibrary
  citation each.

## Open questions

None blocking implementation. The following are deliberate v1 omissions and
land if and when a real user surfaces the need:

- Per-user provider keys.
- Background pre-generation on download.
- Localization of generated insight text.
- Token-budget quotas.
- Multi-replica deployments (current per-doc lock is in-process).
- Calibre plugin readout of insights (parallel to the planned read-only
  progress plugin).

## Decision log

| # | Decision | Rationale |
|---|---|---|
| 1 | Single design covering substrate + both feature pillars, phased rollout. | Substrate decisions dominate everything; book insights v1 is small enough to ship with the substrate. |
| 2 | OpenAI-compatible client with user-supplied `base_url`. | Cloud and local are configuration, not code. |
| 3 | All AI calls in opds-sync, not in-app. | Shared cache, library data is already server-side, jobs survive app death, coalescing. |
| 4 | Server-wide config; per-user opt-in toggle. | Matches the homelab audience; keeps secrets out of Postgres; users still control their data. |
